# Architecture

## Purpose

`admin-template` is a reusable enterprise admin scaffold for Java backend projects. It keeps Ant Design Pro and Umi Max as the application foundation, while adding local conventions for API adapters, CRUD modules, shared business components, i18n, permissions, and AI-assisted development. The authoritative business development rules live in `docs/development-standard.md`.

## Runtime Shape

- `config/config.ts` owns Umi Max plugins, layout, locale, request, OpenAPI, mock, Tailwind, and build settings.
- `config/routes.ts` owns declarative routes and menu entries.
- `src/app.tsx` owns initial user loading, layout runtime behavior, request runtime config, and global error boundary wrapping.
- `src/access.ts` owns access decisions based on `initialState.currentUser`.
- `src/requestErrorConfig.ts` owns request error handling and request interceptors.
- `src/services/openapi/` is generated by the OpenAPI plugin. Do not hand-edit it.

## Page Structure

Business pages should be grouped by domain under `src/pages/<group>/<module>/`. Each page should keep its own types, constants, local service functions, and child components together. The standard example is `src/pages/system/example/`.

Normal CRUD pages should prefer `PageContainer`, `ProTable`, `ModalForm` / `ProForm`, and `ProDescriptions` before falling back to Ant Design base components.

## Shared Components

`src/components/biz/` contains small business-level wrappers for common admin UI patterns:

- `BizStatusTag`
- `BizPermissionButton`
- `BizDictSelect`
- `BizDetailDrawer`
- `BizFormModal`
- `BizToolbar`

These components are intentionally lightweight. They are not a CRUD engine and should not hide page-specific behavior.

## Data Flow

For scaffold examples, page `service.ts` files may return local mock promises. Real business projects should adapt backend responses to the project's internal types through one of these approaches:

- Generated services from OpenAPI under `src/services/openapi/`.
- Per-page `service.ts` that calls Umi Max `request` and normalizes backend wire shapes.
- Response interceptors in `src/requestErrorConfig.ts` for cross-cutting response normalization.
- TanStack Query only when server state needs cache, invalidation, retries, or cross-page coordination.

## Permissions

The current baseline keeps the Ant Design Pro `access.ts` convention. Route access is centralized in `src/access.ts`; the built-in `canAdmin` rule checks `currentUser.access === 'admin'`. `BizPermissionButton` accepts `permissionCode` as a future integration point and renders by default until a real permission adapter is added.

Permission codes should use `<domain>:<module>:<action>`, for example `system:example:create`.

## Internationalization

The default scaffold maintains only `en-US` and `zh-CN`. `config/config.ts` defaults to `en-US` with `baseNavigator: false`, and the header language switcher is hidden automatically when only one locale is available.

## Styling

Use Ant Design Pro defaults first. For page layout, prefer the existing ProComponents surfaces and compact Ant Design spacing. Use Tailwind CSS v4 for layout utilities, antd-style for token-aware styles, CSS Modules for isolated page styles, and Less only for legacy code.

## What Should Stay Stable

- Umi Max configuration shape.
- ProLayout runtime in `src/app.tsx`.
- Request error handling entry point.
- OpenAPI generated service boundary.
- Standard CRUD module directory shape.
